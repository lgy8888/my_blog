                      # 原子性

                      Atomicity

                      事务是一个不可分割的工作单位，要么全部提交，要么全部失败回滚。

                      # 一致性

                      Consistency

                      一致性是指事务执行前后，数据从一个合法性状态变换到另外一个合法性状态。

                      这种状态是语义上的而不是语法上的，跟具体的业务有关

                      什么是合法性状态呢？

                      满足预定的约束的状态就叫做合法的状态

                      通俗一点，这状态是由你自己来定义的（比如满足现实世界中的约束）

                      满足这个状态，数据就是一致的，不满足这个状态，数据就是不一致的

                      如果事务中的某个操作失败了，系统就会自动撤销当前正在执行的事务，返回到事务操作之前的状态

                      如何保证数据的一致性？

                      数据库本身可以保证一部分一致性需求

                      • MySQL数据库支持主键、唯一索引、外键、NOT NULL约束，同时也支持适用CHECK自定义约束
                      • 但是MySQL仅仅支持CHECK语法，实际上即使使用了CHECK子句自定义约束来创建表，但后续插入或更新记录时，MySQL不会检查CHECK子句中的约束是否成立
                        • 虽然CHECK子句在MySQL中无效，但是可以通过定义触发器的方式来自定义一些约束条件来保证数据库中数据的一致性
                      • 如果是SQL Server 或者 Oracle就会在插入或更新时检查是否符合CHECK子句中定义的约束，不成立则拒绝插入或更行

                      更多的一致性需求需要靠业务代码实现

                      • 在更改数据库数据时，进行一致性检查是一个耗费性能的工作
                      • 比如定义一个触发器，每次更新都会校验余额的值是否大于0，或者是更加严格的约束，比如每一笔交易完成后，都需要保证整个系统的越等于所有账户的收入减去所有账户的支出，同时账单表记录以亿为单位，那么这个校验就会耗费很长的时间，每一次付款都会花很多时间

                      # 隔离性

                      Isolation

                      一个事务的执行不能被其他事务干扰

                      一个事务内部的操作及使用的数据对并发的其他事务是隔离的，并发执行的各个事务之间不能互相干扰

                      如何保证事务的隔离性？

                      隔离性是通过锁来保证的。

                      并发事务存在许多问题，脏读、不可重复读、幻读等

                      脏读 (Dirty Reads): 除非事务已提交，否则其他事务不能读取该事务修改但还未提交的数据。在InnoDB中，使用行级锁来防止脏读。

                      不可重复读 (Non-repeatable Reads): 在同一个事务中多次读取同一数据，由于其他事务的修改导致前后读取的结果不一致。通过行级锁和MVCC (多版本并发控制) 来防止不可重复读。

                      幻读 (Phantom Reads): 一个事务读取了一些行之后，另一个并发事务插入了一些新行，然后第一个事务再次读取时看到了这些新行。InnoDB使用间隙锁和Next-Key Locks来防止幻读。

                      # 持久性

                      Durability

                      一个事务一旦被提交，它对数据库中数据的改变就是永久性的，接下来的其他操作和数据库故障不应该对其有任何影响

                      如何保证事务的持久性？

                      持久性是通过redo日志来保证的。

                      数据库在执行写事务的时候，会以mtr为单位记录下一组不可分割的redo日志

                      redo日志在每次提交事务的时候会被刷入磁盘中的日志文件，MySQL后台也有一个线程专门用来每秒日志刷盘

                      数据库崩溃重启之后，可以通过定位到未刷盘的redo日志来进行数据恢复

                      # 事务的概念

                      什么是事务？

                      把需要保证ACID的一个或多个数据库操作称为一个事务(Transaction)

                      事务处理的原则

                      保证所有事务都作为一个工作单元来执行，即使出现了故障，都不能改变这种执行方式。

                      当在一个事务中执行多个操作时，要么所有的事务都被提交( commit )，那么这些修改就永久地保存下来；要么数据库管理系统将放弃所作的所有修改，整个事务回滚( rollback )到最初状态。

                      事务的不同状态

                      事务对应一个或多个数据库操作 ，这些操作执行的不同阶段把事务划分为以下状态：

                      • 活动的（active）
                        • 事务对应的数据库操作正在执行过程中时，该事务处在活动的状态
                      • 部分提交的（partalliy commited）
                        • 当事务中的最后一个操作执行完成，但由于操作都在内存中执行，所造成的影响没有刷新到磁盘时，该事务处在部分提交的状态
                      • 失败的（failed）
                        • 当事务处在活动的或者部分提交的状态时，可能遇到了某些错误（数据库自身的错误、操作系统错误或者直接断电等）而无法继续执行，或者人为的停止当前事务的执行，我们就说该事务处在失败的状态。
                      • 中止的（aborted）
                        • 如果事务执行了半截而变为失败的状态，那么就需要把已经修改的a账户余额调整为未转账之前的金额，换句话说，就是要撤销失败事务对当前数据库造成的影响。
                        • 比如我们转账的事务，当a账户的钱被扣除，但是b账户的钱没有增加时遇到了错误，从而当前事务处在了失败的状态
                        • 书面一点的话，我们把这个撤销的过程称之为回滚。当回滚操作执行完毕时，也就是数据库恢复到了执行事务之前的状态，我们就说该事务处在了中止的状态。
                      • 提交的（committed）
                        • 当一个处在部分提交的状态的事务将修改过的数据都同步到磁盘上之后，我们就可以说该事务处在了提交的状态。

                      只有当事务处于提交的或者中止的状态时，一个事务的生命周期才算是结束了
