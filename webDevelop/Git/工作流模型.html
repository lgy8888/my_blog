                        # 工作流模型

                        # Git Flow

                        # main

                        • 主分支
                        • 一条核心分支，长期存在
                        • 只接受来自hotfix和release的合并请求
                        • 不允许直接Push修改
                        • 该分支部署在生产环境中，保证该分支一直可用
                        • 版本号规则
                          • 主版本（MajorVersion）：主要的功能变化或重大更新
                          • 次版本（MinorVersion）：一些新的功能，改进和更新，不影响现有功能
                          • 修订版本（PatchVersion）：一些小的bug修复，安全漏洞补丁等，通常不会影响现有的功能和接口

                        # hotfix

                        • 热补丁，问题修复分支
                        • 包含了项目某个问题修复的源码，用于修复线上的问题
                        • 修复完成后合并回main分支，合并完会将该分支删除，更新小版本号
                        • 命名规范：hotfix-#issueid-desc

                        # develop

                        • 开发分支
                        • 另一条核心分支，长期存在
                        • 从主线分支中分离出来，包含最新版本的代码，用于开发和测试

                        # feature

                        • 功能分支
                        • 包含了项目某个新功能的代码
                        • 从开发分支中分离出来的
                        • 功能分支中的代码稳定后会以某种方式合并回开发分支中

                        # release

                        • 版本发布分支，预发布分支
                        • 包含了项目最新预发布的代码，用于发布前的测试和验证
                        • 预发布分支的代码稳定后会合并到主分支和开发分支中

                        GitFlow模型适用于团队开发水平适中，有一定的开发流程和规范的团队

                        GitHubFlow模型适用于团队开发水平较高，或一些开源项目

                        # GitHub Flow

                        # Master

                        • 一个长期存在的主分支
                        • 主分支上的代码是可以直接部署到生产环境中
                        • 一般会设置分支保护，禁止团队成员在主分支上进行提交
                        • 团队成员可以从主分支中分理处自己的分支进行开发和测试，然后在本地分支提交代码
                        • 开发完成后发起一个合并请求Pull Request，简称PR
                        • 团队成员对代码进行Review评审
                        • 没问题则将这个PR发布和合并到主分支中

                        # 分支命名

                        • 使用有意义的描述性名称来命名分支
                        • 版本发布分支/Tag：v1.0.0
                        • 功能分支：feature-login-page
                        • 修复分支：hotfix-#issueid-desc

                        # 分支管理

                        • 定期合并已经成功验证的分支，及时删除已经合并的分支
                        • 保持合适的分支数量
                        • 为分支设置合适的管理权限
