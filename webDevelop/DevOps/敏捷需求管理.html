                                    # 敏捷需求管理

                                    # 用户故事

                                    用户故事描述了对客户，系统或软件购买者有价值的功能

                                    用户故事由一下三方面组成

                                    • 一份书面的故事描述卡片，用来做计划和违提示
                                    • 有关故事的交流，用于具体化故事细节
                                    • 测试，哟关于记录和确认故事细节且可用于确定故事何时完成

                                    用户故事是敏捷开发方法中用来定义系统需要提供的功能和作为需求管理条目化的基础

                                    # 描述用户故事

                                    As who, I want what, so that why.

                                    两种故事描述方式

                                    • 作为X（什么用户角色）
                                    • 我希望Y（系统提供什么功能）
                                    • 以便Z（达到什么目的或实现什么价值）

                                    或者

                                    • 为了X（达到什么目的或实现什么价值）
                                    • 作为Y（什么用户角色）
                                    • 我希望Z（系统提供什么功能）

                                    # 角色类型的创建

                                    头脑风暴 -> 整理角色集合 -> 整合角色 -> 提炼角色

                                    # 头脑风暴

                                    • 只创造角色
                                    • 不讨论角色

                                    # 整理角色集合

                                    • 将角色放在一起进行分组
                                    • 有关系的卡片进行重叠

                                    # 整合角色

                                    • 卡片的作者详细描述一下卡片角色

                                    • 背后有什么意义

                                    • 角色究竟代表什么

                                    • 分析重叠卡片之间的联系

                                      • 选择丢弃一个保留一个
                                      • 或者融合成一张新的卡片，旧卡片作为角色属性

                                    # 提炼角色

                                    • 角色特征考虑：
                                      • 操作计算机的熟练程度
                                      • 专业技术水平
                                      • 外语能力
                                      • ……

                                    # 搜集故事

                                    • 用户访谈需要提供比较多的细节

                                      • 不要提封闭式的问题：答案是或者否

                                      • 问题的设置开放一些，让用户多针对这个问题表达他的想法

                                    • 问卷调查的问题需要尽可能考虑用户可能答案是否真的有价值

                                      • 例如：什么功能是你最想看到的（在下方提供选项）

                                    • 故事编写工作坊

                                      • 组织开发人员写故事

                                    # INVEST原则

                                    • 独立性：尽量避免依赖，有依赖也得低优先级依赖高优先级
                                    • 可讨论的：用户故事不应该是有给对需求过分充分的描述信息
                                    • 有价值的：解决一个技术问题，应该描述成对用户由什么价值
                                    • 可估计的：太大或缺乏专业知识，考虑探针的方式，或者拆分用户故事
                                    • 合适大小：太大的故事不好估算，风险也大；太小管理成本高
                                    • 可测试的：有些具体的值可以进行量化

                                    # 用户故事的层级关系

                                    # 用户故事的优先级

                                    最主要的就是要根据业务价值来确定优先级，即这个用户故事能给客户带来多少经济上的收入

                                    还要根据成本，风险，新知识确定优先级

                                    风险：是否能在当前迭代里完成需求，完不成会带来什么问题

                                    新知识会带来成本和风险的增加，未来的迭代里面是否会继续用到新知识

                                    # 确定优先级

                                    # ——渴望度Kano模型

                                    • 当我们购买一件产品时，有哪些功能是我们觉得理所应当有的
                                    • 哪些是我们觉得越多越好的
                                    • 哪些是如果有会令我们觉得兴奋的
                                    • 纵坐标轴代表客户满意度
                                    • 横坐标轴代表产品和功能的贴合成都

                                    例如：冰箱

                                    • 制冷属于冰箱的基本属性，有是应该的，没有客户就要投诉了
                                    • 噪音属于冰箱的线性功能，噪音越高，客户满意度越差，噪音越低，客户满意度越高
                                    • 冰箱可以播放音乐，没有的话也没什么，有的话客户会觉得非常满意

                                    # 估算方法

                                    估算故事点数

                                    • 开发人员对同一个故事写一个数字

                                      • 若都差不多，就取平均值
                                      • 若相差较大，就让数最大的和最小的分别说一下理由，交换意见后重新评估

                                    • 专家意见，类比，分解

                                    # 分解用户故事的方法

                                    数据边界分解

                                    按照用户故事所支持的数据边界来分解大型用户故事

                                    例如：资产分解为流动和非流动，或者具体为房产、现金、证券等
