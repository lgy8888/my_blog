                        # 规划与设计

                        # 增量式交付

                        • 起步于粗糙的产品原型/框架
                          • MVP -> 最小可用产品
                        • 细节功能不完善，但可以验证用户的整体使用场景
                          • 行走的骨骼 ->没有人形，但是具备了 人的所有基础框架
                        • 每增加一点功能，用户都能体会到价值的提升
                          • 持续改进 -> 一次把糖果都给小朋友和每次给一颗的区别
                        • 与用户一同寻找最终的产品
                          • 用户能从早期的增量，更加理解后面的需求

                        # 可视化产品研发

                        # 影响地图

                        使用可视化的方式，建立商业目标与产品功能的联系，并将联系背后的假设可视化出来

                        # 四个问题

                        通过四个问题达成统一的认知

                        • WHY：地图的中心描述了我们为什么做这些？也就是我们试图达成的目标是什么
                          • 可以利用5WHY工具，找到的目标要符合SMART原则
                          • 例如：100万玩家
                        • WHO：地图的第一层描述了谁能产生所需要的效果？谁会阻碍它？谁是我们产品的消费者或用户？谁会被他影响？也就是哪些角色回影响结果
                          • 所有和目标有关联的角色都要进行考虑
                          • 例如：玩家
                        • HOW：地图的第二层描述了角色的行为是怎么样被改变的？他们怎样说明我们达成目标？他们怎样帮助或妨碍我们取得成功？也就是我们想要产生什么影响
                          • 例如：玩家可以通过邀请朋友的方式来帮助实现100万玩家的目标
                        • WHAT：地图的第三层描述了我们可以做什么来支持第二层中所列出的影响的实现
                          • 例如：玩家邀请朋友可以通过半自动化的形式来邀请

                        # 对复杂任务关系进行分析拆分

                        • 可以做什么？
                        • 目标是什么？
                        • 影响地图的会议需要有严格的时间盒，当关键的想法已经出现在地图上，团队已经达成目标并且确定了最小和最快路径，暂时想不出更好的替代方案时，影响地图会议就可以结束

                        # 用户故事地图

                        • 透过可视化的方式，建立用户场景与技术规格之间的联系，并辅助团队有效沟通

                        • 让用户在产品还没有做出来之前就可以体验到产品所提供的功能，并以二维结构呈现产品的主线功能和辅助功能，帮助设计者决策

                        # 用户故事地图使用的三个步骤

                        • 广度优先：从左到右讲述用户经历的重要步骤，要体现交互最好
                        • 深度优先：针对每个主流程进行细化
                        • 版本规划：确定MVP

                        # 可视化研发过程

                        影响地图+用户故事地图产出用户故事+功能点

                        影响地图实现从业务目标到功能规划，用户故事地图实现从功能规划到功能实现
