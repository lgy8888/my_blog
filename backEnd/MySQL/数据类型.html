                                                                  # 数据类型

                                                                  MySQL数据库三种常见的数据类型

                                                                  • 字符串类型
                                                                  • 数字类型
                                                                  • 日期类型

                                                                  这三种常见的数据类型都属于传统关系型设计

                                                                  # 字符串类型

                                                                  MySQL 数据库的字符串类型有 CHAR、VARCHAR、BINARY、BLOB、TEXT、ENUM、SET。

                                                                  不同的类型在业务设计、数据库性能方面的表现完全不同，其中最常使用的是 CHAR、VARCHAR。

                                                                  # VARCHAR

                                                                  # 语法

                                                                  VARCHAR(max_length)
                                                                  • max_length：数值，指定此列的最大字符数，默认为255
                                                                  • VARCHAR => VARCHAR(255)

                                                                  # VARCHAR长度

                                                                  • VARCHAR允许的最大长度65535 —— > 也是MySQL一行的最大值
                                                                    • VARCHAR前2个字节用于记录长度前缀，所以设置VARCHAR最大长度的时候最大只能设置65532 字节
                                                                    • 实际存储的字符串超过255字节，VARCHAR会用2个字节来记录实际字符串的长度，否则用1个字节来记录
                                                                      • 一个字节有8位，最多表示255个字节
                                                                      • 两个字节最多记录到长度65535
                                                                  • VARCHAR存储的数据单位是字节，VARCHAR指定最大长度的单位是字符（也就是括号里面数值的单位）
                                                                    • VARCHAR中能存储多少个字符取决于数据库使用的字符集

                                                                  latin1一个字符对应一个字节 ——> 最大：VARCHAR(65532)

                                                                  utf8一个字符最多占用三个字节 ——> 最大：VARCHAR(65535 / 3 = 21845)

                                                                  # VARCHAR截断

                                                                  插入的长度超过VARCHAR设置的长度怎么办

                                                                  • 如果超过的部分只有空格，多余的空格会被删掉

                                                                    1 row(s) affected, 1 warning(s): 1265 Data truncated for column 'v' at row 1

                                                                  • 如果超出的部分有数据，报错

                                                                    Error Code: 1406. Data too long for column 'v' at row 1

                                                                  # CHAR

                                                                  # 语法

                                                                  CHAR(length)
                                                                  • length：数值，指定了此列的字符数，默认为1
                                                                  • CHAR => CHAR(1)
                                                                  • length可以是0到255之间的任何值

                                                                  # CHAR填充

                                                                  CHAR是定长的，如果写入的实际数据没有那么长呢？

                                                                  • 写入 CHAR 列中的字符串的长度小于指定的字符长度，MySQL 会在源字符串后**填充空格**一直到指定的长度。

                                                                  • 读取 CHAR 列中的值时，MySQL 会**删除后面的空格**。

                                                                  • 由于 CHAR 数据类型采用固定的长度进行存储，因此 CHAR 的性能要比 VARCHAR 更好

                                                                  # CAHR性能

                                                                  为什么使用固定列存储性能就会更好呢？

                                                                  • 查询效率：
                                                                    • 因为CHAR类型字段总是占据固定长度的空间，这使MySQL可以更快速地定位和读取数据
                                                                    • 数据库引擎可以直接通过偏移量来找到需要的数据，不需要进行额外的计算和处理。
                                                                    • VARCHAR类型存储的字符串长度可变，需要**额外的存储空间来保存长度信息**，以及更多的处理逻辑来定位和处理逻辑，需要根据长度来计算偏移量。
                                                                  • 存储空间占用：
                                                                    • 虽然CHAR在读取上有性能优势，但是也会占用更多的存储空间，因为分配的空间是固定的，但是实际上可能一个字符串并不会用到这么长的空间

                                                                  相当于CHAR是用空间换时间

                                                                  # CHAR的空格

                                                                  CHAR会在插入字符串时候在后面填充空格到满，读取字符串的时候将字符串后面的空格删除

                                                                  • 删除的空格不会区分用户写入还是系统填充，在后面的空格都会被删掉
                                                                  • 如果将CHAR列设置为UNIQUE，可能回让只有尾部有空格差异的值无法同时存入，但是VARCAHR就不会存在这样的问题。

                                                                  插入少于长度的值会填充，超出长度呢？

                                                                  插入超过自身字段长度的值，MySQl会直接报错

                                                                  INSERT INTO test_char VALUES('Steven');

                                                                  ERROR 1406 (22001): Data too long for column 'name' at row 1

                                                                  # 字符集

                                                                  # 字符集基础

                                                                  在表结构设计中，除了将列定义为 CHAR 和 VARCHAR 用以存储字符以外，还需要额外定义字符对应的字符集

                                                                  每种字符在不同字符集编码下，对应着不同的二进制值

                                                                  • 常见的字符集有 GBK、UTF8，通常推荐把默认字符集设置为 UTF8。

                                                                  推荐把 MySQL 的默认字符集设置为 UTF8MB4，否则，某些 emoji 表情字符无法在 UTF8 字符集下存储

                                                                  • 比如 emoji 笑脸表情，对应的字符编码为 0xF09F988E
                                                                  • 强行在UTF-8的列上插入emoji表情字符，MySQL会报错
                                                                  ERROR 1366 (HY000): Incorrect string value: '\xF0\x9F\x98\x8E' for column 'a' at row 1

                                                                  包括 MySQL 8.0 版本在内，字符集默认设置成 UTF8MB4，8.0 版本之前默认的字符集为 Latin1

                                                                  因为不同版本默认字符集的不同，你要显式地在配置文件中进行相关参数的配置

                                                                  [mysqld]
character-set-server = utf8mb4
...

                                                                  不同的字符集，CHAR(N)、VARCHAR(N) 对应最长的字节也不相同

                                                                  • 比如 GBK 字符集，1 个字符最大存储 2 个字节，UTF8MB4 字符集 1 个字符最大存储 4 个字节。

                                                                  从底层存储内核看，在多字节字符集下，CHAR 和 VARCHAR 底层的实现完全相同，都是变长存储！

                                                                  • CHAR(1) 既可以存储 1 个 ‘a’ 字节，也可以存储 4 个字节的 emoji 笑脸表情，因此 CHAR 本质也是变长的

                                                                  鉴于目前默认字符集推荐设置为 UTF8MB4

                                                                  所以在表结构设计时，可以把 CHAR 全部用 VARCHAR 替换，底层存储的本质实现一模一样。

                                                                  # 修改字符集

                                                                  ALTER TABLE emoji_test CHARSET utf8mb4;

                                                                  将表的字符集修改为 UTF8MB4，下次新增列时，若不显式地指定字符集，新列的字符集会变更为 UTF8MB4，但对于已经存在的列，其默认字符集并不做修改

                                                                  # BIT

                                                                  BIT 数据类型被用来存储二进制的位值

                                                                  我们通常使用 BIT 数据类型的列存储状态值，比如布尔值等

                                                                  BIT 数据类型也是易于扩展的。

                                                                  # 语法

                                                                  BIT(M)
                                                                  • BIT(M) 允许存储 M 位值。 M 的取值范围是 1 到 64。
                                                                  • 如果不指定 M，那么它的默认值为 1。 BIT 等效于 BIT(1)。

                                                                  # 易拓展

                                                                  使用BIT数据类型存储位字段值时，可以根据需要设置固定的位数

                                                                  假设有一个表格存储用户权限信息，每个用户的权限可以用一个位字段表示

                                                                  初始情况下，可能只需要存储3个权限，所以创建表格时可以定义一个BIT(3)的列。例如：

                                                                  CREATE TABLE user_permissions (
  user_id INT,
  permissions BIT(3)
);

                                                                  每个用户的权限由一个3位的二进制值表示。

                                                                  如果应用程序需要增加一个新的权限，那么可以很容易地扩展BIT字段的位数。

                                                                  只需修改表格结构，将BIT(3)改为BIT(4)，以便能够存储更多的位字段。

                                                                  ALTER TABLE user_permissions
MODIFY COLUMN permissions BIT(4);

                                                                  这样，就能够存储更多的权限信息，而不需要改动其他部分的存储结构。这种扩展性使得你能够在需要时灵活地调整位字段数目，而不会对整个数据库的结构造成过多的影响。

                                                                  此外，BIT数据类型还支持位操作，比如AND、OR、XOR等逻辑运算，这使得在进行位操作时能够更加高效地读取和更新位字段的值。

                                                                  这对于频繁进行位操作或者位字段数目可能会发生变化的场景下非常有优势。

                                                                  # 实例

                                                                  BIT列适合用来存储状态值

                                                                  例子：用BIT列来存储订单状态值

                                                                  创建一个订单状态表 order_state，表中包含了一个用来存储订单状态的 state 列，定义好订单的状态信息

                                                                  状态状态十进制值状态二进制值
                                                                  待支付0000
                                                                  已支付1001
                                                                  待发货2010
                                                                  已发货3011
                                                                  已完成4100

                                                                  二进制值最大位数是 3 位，那么 state 列的数据类型可以使用 state BIT(3)。

                                                                  CREATE TABLE order_state (
  order_id INT NOT NULL PRIMARY KEY,
  state BIT(3) NOT NULL
);

                                                                  插入BIT值

                                                                  可以使用 3, b'011', B'011', 0b011 其中的任意一个

                                                                  INSERT INTO order_state (order_id, state)
VALUES (1, 3),
       (2, b'011'),
       (3, B'011'),
       (4, 0b011);

                                                                  # ENUM

                                                                  ENUM 是一个字符串的列表，它定义了一个列中允许的值，列的值只能是创建列时定义的允许值列表中的的一个。

                                                                  MySQL ENUM 数据类型列适合存储状态和标识等有限数量的固定值的数据。

                                                                  MySQL ENUM 数据类型具有以下优点：

                                                                  • 列值的可读性更强。
                                                                  • 紧凑的数据存储。MySQL 存储 ENUM 时只存储枚举值对应的数字索引 (1, 2, 3, …)。

                                                                  # 语法

                                                                  ENUM ('v1', 'v2', ..., 'vn')
                                                                  • ENUM 是一个关键字，用来声明一个枚举类型。
                                                                  • v1 到 vn 是此 ENUM 类型的可选项列表，使用 ENUM 类型的列的值只能是上面值中的其中一个。
                                                                  • 枚举值只能是字符串

                                                                  定义ENUM列

                                                                  CREATE TABLE table_name
(col_name ENUM ('v1','v2', ..., 'vn'));

                                                                  ENUM 数据类型中可以拥有多个枚举值。

                                                                  但是，将枚举值的数量保持在 20 以下是一种很好的做法。

                                                                  # 使用规则

                                                                  • 在使用时，可以指定具体的ENUM值，也可以直接使用索引来指定
                                                                  • 插入时如果没有给ENUM列指定值，那么默认会插入索引为1的ENUM值
                                                                  • 在非严格 SQL 模式下，如果在 ENUM 列中插入无效值，MySQL 将使用带有数字索引 0 的空字符串 '' 进行插入。
                                                                  • 如果启用了严格 SQL 模式，尝试插入无效 ENUM 值将导致错误。
                                                                  • 如果将 ENUM 列定义为可为空的列，则该列可以接受 NULL 值。

                                                                  # 实例

                                                                  # 创建

                                                                  假设有个订单，订单中有四种状态

                                                                  状态名状态值
                                                                  未支付Unpaid
                                                                  已支付Paid
                                                                  已发货Shipped
                                                                  已完成Completed
                                                                   CREATE TABLE orders (    
     id INT PRIMARY KEY AUTO_INCREMENT,   
     title VARCHAR(255) NOT NULL,    
     state ENUM('Unpaid', 'Paid', 'Shipped', 'Completed') NOT NULL
 );
                                                                  • state 列为 ENUM 数据类型，并且它将只接受四个值: Unpaid, Paid, Shipped 和 Completed。
                                                                  • 同时，按照列定义时的顺序，Unpaid, Paid, Shipped 和 Completed 的索引分别为 1, 2, 3, 4。
                                                                  # 插入
                                                                  INSERT INTO orders(title, state)
VALUES ('Apples', 'Paid');

                                                                  INSERT INTO orders(title, state)
VALUES ('Apples', 2);
                                                                  • 插入的时候可以使用ENUM的值或者索引，二者都是正确的
                                                                  INSERT INTO orders(title)
VALUES ('Oranges');
                                                                  • 如果没有赋值，MySQL会默认插入第一个ENUM成员，也就是unpaid
                                                                  +----+---------+-----------+
| id | title   | state     |
+----+---------+-----------+
|  1 | Apples  | Paid      |
|  2 | Apples  | Paid      |
|  3 | Oranges | Unpaid    |
+----+---------+-----------+

                                                                  # 缺点

                                                                  MySQL ENUM 带来一些好处，比如可读性和存储效率，但是它也有以下缺点：

                                                                  1. 更改枚举成员需要使用ALTER TABLE语句重建整个表，这在资源和时间方面都是昂贵的。
                                                                  2. 获取完整的枚举列表很复杂，因为您需要访问 information_schema 数据库：
                                                                  SELECT column_type
FROM information_schema.COLUMNS
WHERE TABLE_NAME = 'orders'
AND COLUMN_NAME = 'state';
                                                                  1. 因为 ENUM 它不是 SQL 标准的，因此，移植到其他 RDBMS 可能是一个问题。
                                                                  2. 枚举列表是不可重用的。例如，上面 orders 表中的状态枚举值，不能重用到新建的其他表上。
                                                                  3. 枚举值是字符串，不能包含更多的信息。比如我们需要在每个订单状态上添加超时属性。

                                                                  # 业务表结构设计

                                                                  # 用户性别设计

                                                                  设计表结构时，你会遇到一些固定选项值的字段。

                                                                  • 例如，性别字段（Sex），只有男或女；又或者状态字段（State），有效的值为运行、停止、重启等有限状态。
                                                                  # 使用INT类型

                                                                  大多数开发人员喜欢用 INT 的数字类型去存储性别字段

                                                                  CREATE TABLE `User` (
  `id`  bigint NOT NULL AUTO_INCREMENT,
  `sex` tinyint DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB；

                                                                  其中，tinyint 列 sex 表示用户性别，但这样设计问题比较明显。

                                                                  • 表达不清：在具体存储时，0 表示女，还是 1 表示女呢？每个业务可能有不同的潜规则；
                                                                  • 脏数据：因为是 tinyint，因此除了 0 和 1，用户完全可以插入 2、3、4 这样的数值，最终表中存在无效数据的可能，后期再进行清理，代价就非常大了。
                                                                  # 使用ENUM类型

                                                                  使用 ENUM 字符串枚举类型，只允许有限的定义值插入。

                                                                  Create Table: CREATE TABLE `User` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `sex` enum('M','F') COLLATE utf8mb4_general_ci DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB

                                                                  如果将参数 SQL_MODE 设置为严格模式，插入非定义数据就会报错

                                                                  由于类型 ENUM 并非 SQL 标准的数据类型，而是 MySQL 所独有的一种字符串类型。抛出的错误提示也并不直观

                                                                  # 使用CHECK约束字段

                                                                  这样的实现总有一些遗憾，主要是因为MySQL 8.0 之前的版本并没有提供约束功能

                                                                  自 MySQL 8.0.16 版本开始，数据库原生提供 CHECK 约束功能，可以方便地进行有限状态列类型的设计

                                                                  Create Table: CREATE TABLE `User` (
  `id` 	bigint NOT NULL AUTO_INCREMENT,
  `sex` char(1) COLLATE utf8mb4_general_ci DEFAULT NULL,
  PRIMARY KEY (`id`),
  CONSTRAINT `user_chk_1` CHECK (((`sex` = _utf8mb4'M') or (`sex` = _utf8mb4'F')))
) ENGINE=InnoDB

                                                                  mysql> INSERT INTO User VALUES (NULL,'M');
Query OK, 1 row affected (0.07 sec)

mysql> INSERT INTO User VALUES (NULL,'Z');
ERROR 3819 (HY000): Check constraint 'user_chk_1' is violated.
                                                                  • 第 8 行的约束定义 user_chk_1 表示列 sex 的取值范围，只能是 M 或者 F
                                                                  • 同时，当 15 行插入非法数据 Z 时，你可以看到 MySQL 显式地抛出了违法约束的提示

                                                                  # 账密存储设计

                                                                  在数据库表结构设计时，千万不要直接在数据库表中直接存储密码，一旦有恶意用户进入到系统，则面临用户数据泄露的极大风险

                                                                  • 比如金融行业，从合规性角度看，所有用户隐私字段都需要加密
                                                                  • 甚至业务自己都无法知道用户存储的信息（隐私数据如登录密码、手机、信用卡信息等）
                                                                  # MD5算法加密

                                                                  利用MD5 算法不可逆，通过函数 MD5 加密存储隐私数据。然而，MD5 加密后的值是固定的，因此，可以对 MD5 进行暴力破解

                                                                  # MD5加盐

                                                                  在设计密码存储使用，还需要加盐（salt）

                                                                  每个公司的盐值都是不同的，因此计算出的值也是不同的

                                                                  固定盐值的加密算法存在三个主要问题：

                                                                  • 若 salt 值被（离职）员工泄漏，则外部黑客依然存在暴利破解的可能性；
                                                                  • 对于相同密码，其密码存储值相同，一旦一个用户密码泄漏，其他相同密码的用户的密码也将被泄漏；
                                                                  • 固定使用 MD5 加密算法，一旦 MD5 算法被破解，则影响很大。
                                                                  # 动态盐+非固定加密算法

                                                                  密码列存储在数据库中的格式为

                                                                  $salt$cryption_algorithm$value

                                                                  • $cryption_algorithm：表示加密的算法
                                                                    • 如 v1 表示 MD5 加密算法，v2 表示 AES256 加密算法，v3 表示 AES512 加密算法等。
                                                                  • $value：表示加密后的字符串。

                                                                  # DECIMAL

                                                                  DECIMAL 数据类型是定点数数据类型，用来存储精确的数值，比如账务金额等

                                                                  底层实现上，MySQL 使用二进制形式存储该类型的值。

                                                                  # 语法

                                                                  DEC, NUMERIC, FIXED 是 DECIMAL 的同义词。

                                                                  DECIMAL[(M[,D])] [UNSIGNED] [ZEROFILL]
                                                                  • M 是总的位数，不包含小数点和正负号。
                                                                  • D 是小数部分的位数。如果 D 为 0 则表示没有小数部分。当 D 省略时，默认值为 0。
                                                                  • UNSIGNED 属性表示数值是无符号的。无符号的数字不能是负数。
                                                                  • ZEROFILL 属性表示当整数部分位数不足时，用整数的左侧用 0 填充。带有 ZEROFILL 的列将自动具有 UNSIGNED 属性。这和 INT 数据类型一样。

                                                                  如何理解位数属性M和小数点后位数属性D

                                                                  amount DECIMAL(9, 4);

                                                                  amount 列的值的范围是从 -99999.9999 到 99999.9999

                                                                  • 9位数值里面后四位是小数位
                                                                  amount DECIMAL(9);

                                                                  amount 列的值的范围是从 -999999999 到 999999999

                                                                  • 9位数值全部是整数位

                                                                  如何插入？

                                                                  • 如果插入的数字的整数部分长度超过M-D，就会报错

                                                                  • 如果小数部分的长度超过D，不会报错

                                                                  • 整个数字长度超过M不会报错

                                                                  # 资金字段设计

                                                                  在用户余额、基金账户余额、数字钱包、零钱等的业务设计中

                                                                  由于字段都是资金字段，通常程序员习惯使用 DECIMAL 类型作为字段的选型，因为这样可以精确到分

                                                                  • 如：DECIMAL(8,2)
                                                                  CREATE TABLE User (
  userId BIGINT AUTO_INCREMENT,
  money DECIMAL(8,2) NOT NULL,
  ......
)

                                                                  在海量互联网业务的设计标准中，并不推荐用 DECIMAL 类型，而是更推荐将 DECIMAL 转化为 整型类型

                                                                  也就是说，资金类型更推荐使用用分单位存储，而不是用元单位存储。如1元在数据库中用整型类型 100 存储

                                                                  why？

                                                                  • DECIMAL是变长字段，需要定义总位数，而资金字段通常是很难给出确切的长度的

                                                                    • 用户的金额至少要存储百亿级别
                                                                    • 统计局GDP金额字段可能需要存数十万亿级别
                                                                    • 用DECIMAL定义不好统一

                                                                  • DECIMAL底层是二进制实现的编码方式，计算效率远不如整型来的高效

                                                                  因此推荐使用BIGINT来存储金额相关的字段

                                                                  字段存储的时候采用分为单位存储，BIGINT可以达到千兆级别的金额（1兆 = 1万亿）

                                                                  小数点中的数据完全可以交由前端进行处理并展示。作为数据库本身，只要按分进行存储即可

                                                                  BIGINT存储金额字段优点

                                                                  • 所有金额相关字段都是定长字段，占用8个字节，存储高效
                                                                  • 直接通过整型计算，效率更高

                                                                  # 变长存储和定长存储

                                                                  # 碎片空间

                                                                  数据库中记录的存储方式

                                                                  • 若发生更新，记录 1 原先的空间无法容纳更新后记录 1 的存储空间
                                                                  • 因此，这时数据库会将记录 1 标记为删除，寻找新的空间给记录1使用
                                                                  • 记录 1 表示的就是原先记录 1 占用的空间
                                                                  • 而这个空间后续将变成碎片空间，无法继续使用，除非人为地进行表空间的碎片整理

                                                                  # 定长存储

                                                                  # 优势

                                                                  1. 性能优势： 定长存储通常更有利于性能
                                                                    • 每个字段都占据固定的存储空间。在读取和写入数据时，数据库**引擎可以更轻松地直接定位到存储位置，无需解析变长字段**。
                                                                  2. 查询速度： 定长存储对于执行一些特定类型的查询可能更有效率，特别是涉及到按照字段位置或长度进行计算的查询。
                                                                  3. 避免碎片化： 使用定长存储可以减少数据碎片化的可能性
                                                                    • 因为每个记录都占据相同的存储空间。这有助于**防止由于数据的不断插入和删除而导致的碎片化问题**。
                                                                  4. 更易于管理： 定长存储的结构**通常更容易管理**，因为不同记录之间没有变长字段的长度差异，这简化了数据库的维护和优化。

                                                                  # 缺点

                                                                  1. 空间利用不高： 定长存储可能会浪费一些存储空间
                                                                    • 特别是对于那些变长字段在大多数情况下都不使用其最大长度的情况。
                                                                  2. 灵活性差： 当需要存储变长数据时，定长存储可能不够灵活
                                                                    • 因为它需要为每个字段分配足够的空间，这可能导致一些浪费。
                                                                  3. 处理变长数据困难： 对于经常变动长度的字段，使用定长存储可能会使数据的插入和更新变得更为复杂。

                                                                  # 变长存储

                                                                  # 优点

                                                                  1. 空间利用效率高： 变长存储可以更灵活地使用存储空间，只占用实际数据所需的空间，避免了定长存储可能存在的空间浪费问题。
                                                                  2. 适应不同长度的数据： 变长存储更适合**存储长度不固定的数据**
                                                                    • 如文本字段。这样可以节省空间，并且更灵活地处理各种数据类型。
                                                                  3. 易于扩展和维护： 当需要**修改字段长度或添加新字段时，变长存储更为灵活**，不需要调整整个表结构，因此更容易进行数据库的维护和演进。
                                                                  4. 支持索引效果好： 在某些情况下，变长存储**对于建立索引可能更为高效**
                                                                    • 因为索引的大小取决于存储的实际数据长度。

                                                                  # 缺点

                                                                  1. 性能损耗： 由于变长存储需要额外的元数据来存储字段长度信息，读取和写入数据时可能涉及更多的计算操作，导致性能相对较低，尤其在处理大量数据时。
                                                                  2. 碎片化问题： 变长存储可能会导致数据碎片化，因为在删除变长字段中的数据时，会留下一些空间无法被其他数据使用，可能需要进行定期的碎片整理操作。
                                                                  3. 难以处理大型对象： 对于大型对象（如大型文本或二进制数据），变长存储可能需要额外的管理和维护，因为这些对象的长度不断变化可能会导致性能问题。
                                                                    • 动态调整存储空间、维护元数据、处理碎片化等操作可能使数据库系统的维护变得更加复杂
                                                                  4. 数据访问相对复杂： 变长存储可能需要更多的计算来定位和读取数据，相对于定长存储，处理变长存储的数据可能会更为复杂。

                                                                      • 00-69范围内的年份值转为2000-2069

                                                                  # DATETIME

                                                                  DATETIME可以用来存储包含日期和时间的值

                                                                  DATETIME的数据格式为：

                                                                  YYYY-MM-DD hh:mm:ss

                                                                  默认情况下，DATETIME的值从1000-01-01 00:00:00 到 9999-12-31 23:59:59

                                                                  一个DATETIME值固定占用 8 个字节

                                                                  • 前4个字节存储日期部分，以自1970年1月1日以来的秒数表示（使用32位的整数存储）。
                                                                  • 后4个字节存储时间部分，以一天的秒数表示（同样使用32位的整数存储）。

                                                                  从 MySQL 5.6 版本开始，DATETIME 类型支持毫秒，DATETIME(N) 中的 N 表示毫秒的精度。

                                                                  • 格式：YYYY-MM-DD hh:mm:ss[.fraction]
                                                                  • 例如：2015-12-20 10:01:00.999999

                                                                  使用毫秒时，DATETIME需要更多的存储空间

                                                                  小数秒精度存储（字节）
                                                                  00
                                                                  1、21
                                                                  3、42
                                                                  5、63
                                                                  • 2015-12-20 10:01:00.999999需要9个字节
                                                                    • 2015-12-20 10:01:00占5个字节
                                                                    • .999999占3个字节
                                                                  • 2015-12-20 10:01:00.9需要6个字节
                                                                    • 2015-12-20 10:01:00占5个字节
                                                                    • .9占1个字节

                                                                  请注意，在 MySQL 5.6.4 之前， DATETIME 值需要 8 个字节的存储空间，而不是 5 个字节。

                                                                  # TIMESTAMP

                                                                  MySQL 提供了另一种类似于 DATETIME 的 时间数据类型 TIMESTAMP 。

                                                                  二者的区别

                                                                  TIMESTAMP需要4个字节，而DATETIME需要8个字节

                                                                  二者都需要额外字节来存储小数秒

                                                                  TIMESTAMP值的范围从1970-01-01 00:00:01 UTC到2038-01-19 03:14:07 UTC

                                                                  • 要存储超过 2038 年的时间值，则应使用 DATETIME 代替 TIMESTAMP

                                                                  TIMESTAMP实际存储的内容是时间戳，也就是 1970-01-01 00:00:00 到现在的毫秒数

                                                                  TIMESTAMP优点

                                                                  TIMESTAMP 最大的优点是可以带有时区属性，因为它本质上是从毫秒转化而来

                                                                  如果你的业务需要对应不同的国家时区，那么类型 TIMESTAMP 是一种不错的选择。

                                                                  • 比如新闻类的业务，通常用户想知道这篇新闻发布时对应的自己国家时间，那么 TIMESTAMP 是一种选择。
                                                                  • 有些国家会执行夏令时。根据不同的季节，人为地调快或调慢 1 个小时，带有时区属性的 TIMESTAMP 类型本身就能解决这个问题

                                                                  # 日期时间与时间戳

                                                                  时间戳涉及到不同时区，而日期时间使用的都是协调时区，就是+00:00

                                                                  # MySQL的时区

                                                                  MySQL TIMESTAMP 的时区信息以 UTC 值存储。但是，MySQL 将 DATETIME 值按原样存储，没有时区

                                                                  设置当前连接的时区设置为 +00:00

                                                                  SET time_zone = '+00:00';

                                                                  CREATE TABLE timestamp_n_datetime (
    id INT AUTO_INCREMENT PRIMARY KEY,
    ts TIMESTAMP,
    dt DATETIME
);

VALUES(NOW(),NOW());

SELECT ts, dt

+---------------------+---------------------+

+---------------------+---------------------+

                                                                  可以看到，在**+00:00**时区下两个值相同

                                                                  总结

                                                                  • 更改时区时， TIMESTAMP 列以 UTC 存储日期和时间值，该列的值会根据新时区进行调整。
                                                                  • 这意味着使用 TIMESTAMP 数据来存储日期和时间值，在将数据库移动到位于不同时区的服务器时应该认真考虑

                                                                  # 业务表设计

                                                                  # 如何选择

                                                                  在做表结构设计时，对日期字段的存储，开发人员通常会有 3 种选择：DATETIME、TIMESTAMP、INT

                                                                  用INT

                                                                  INT 类型就是直接存储 ‘1970-01-01 00:00:00’ 到现在的毫秒数，本质和 TIMESTAMP 一样，因此用 INT 不如直接使用 TIMESTAMP。

                                                                  更重要的是，在后期运维和数据分析时，使用 INT 存储日期，是会让 DBA 和数据分析人员发疯的，INT的可运维性太差

                                                                  用TIMESTAMP

                                                                  用类型 TIMESTEMP 存储日期，因为类型 TIMESTAMP 占用 4 个字节，比 DATETIME 小一半的存储空间。

                                                                  但若要将时间精确到毫秒，TIMESTAMP 要 7 个字节，和 DATETIME 8 字节差不太多

                                                                  另一方面，现在距离 TIMESTAMP 的最大值‘2038-01-19 03:14:07’已经很近，这是需要好好思考的问题。

                                                                  不如DATETIME

                                                                  总的来说，我建议你使用类型 DATETIME。 对于时区问题，可以由前端或者服务这里做一次转化，不一定非要在数据库中解决。

                                                                  # TIMESTAMP的性能问题

                                                                  TIMESTAMP 的上限值 2038 年很快就会到来，那时业务又将面临一次类似千年虫的问题

                                                                  如果使用默认的操作系统时区，则每次通过时区计算时间时，要调用操作系统底层系统函数 __tz_convert()，而这个函数需要额外的加锁操作，以确保这时操作系统时区没有修改

                                                                  所以，当大规模并发访问时，由于热点资源竞争，会产生两个问题。

                                                                  • 性能不如 DATETIME： DATETIME 不存在时区转化问题。
                                                                  • 性能抖动： 海量并发时，存在性能抖动问题。

                                                                  为了优化 TIMESTAMP 的使用，强烈建议你使用显式的时区，而不是操作系统时区。

                                                                  • 比如在配置文件中显示地设置时区，而不要使用系统时区：
                                                                  • 使用显示的设置时区可以提升百分之五十左右的性能
                                                                  # 表结构设计规范

                                                                  在做表结构设计规范时，每张业务核心表都增加一个 DATETIME 类型的 last_modify_date 字段，并设置修改自动更新机制， 即标识每条记录最后修改的时间。

                                                                  这样设计的好处是： 用户可以知道每个用户最近一次记录更新的时间，以便做后续的处理。

                                                                  • 在电商的订单表中，可以方便对支付超时的订单做处理
                                                                  • 在金融业务中，可以根据用户资金最后的修改时间做相应的资金核对等。

                                                                  # YEAR

                                                                  YEAR数据类型用来存储年份值

                                                                  只占1个字节，范围从1901到2155，还有0000

                                                                  YEAR列可以接受多种输入格式

                                                                  • 4 位数字的年份值，从 1901 to 2155。
                                                                  • 4 位数字的年份值的字符串形式，从 '1901' 到 '2155'。
                                                                  • 2 位数字的年份值，从0到99。按如下规则转换为 4 位数年份：
                                                                    • 1 到 69 转换为 2001 到 2069。
                                                                    • 70 到 99 转换为 1970 到 1999。
                                                                    • 0 转换为 0000。
                                                                  • 2 位数字的年份值的字符串形式，从'0'到'99'。按如下规则转换为 4 位数年份：
                                                                    • '0' 到 '69' 转换为 2000 到 2069。
                                                                      • 字符串'0'转换为2000而不是0000
                                                                    • '70' 到 '99' 转换为 1970 到 1999。

                                                                  若未启用严格 SQL 模式，MySQL 会将无效的 YEAR 值转为 0000。在严格 SQL 模式下，插入无效的 YEAR 值时会产生错误。

                                                                  # 语法

                                                                  column_name YEAR

                                                                  在 MySQL 8.0.19 之前的版本中，YEAR 可以使用显示宽度属性，如： YEAR(2)。

                                                                  现在显示宽度属性已经不再被推荐使用

                                                                  并且 MySQL 8 以后也不支持 YEAR(2) 了。直接使用 YEAR。

                                                                  关系型的结构化存储存在一定弊端

                                                                  • 它需要预先定义好所有的列以及列对应的类型
                                                                  • 业务发展过程中，可能需要扩展单个列的描述功能
                                                                  • 使用JSON类型可以大同关系型和非关系型数据存储的界限，为业务提供更好的架构选择

                                                                  JSON和普通的字符串有什么区别

                                                                  JSON表面上看就是一串很大的字符串，但本质上，JSON是一种新的数据类型

                                                                  • 有自己的存储格式，还能在每个对应的字段上创建索引，做特定的优化

                                                                  如果要在生产环境中使用 JSON 数据类型，强烈推荐使用 MySQL 8.0 版本。

                                                                  # 表设计

                                                                  在数据库中，JSON类型适合存储一些修改较少，相对静态的数据

                                                                  CREATE TABLE user_login(
	user_id BIGINT PRIMARY KEY,
    login_info JSON
)

                                                                  由于当前业务的登陆方式越来越多样化，同一账户支持手机/邮箱/微信/QQ登录，甚至微博，GitHub等

                                                                  所以，可以用JSON来存储用户的登录信息

                                                                  SET @a = '{
	"cellphone" : "18088888888",
	"wxchat" : "LGY888888",
	"QQ" : "3179999999"
}'
INSERT INTO user_login VALUES (1, @a);

                                                                  SET @b = '{
	"cellphone" : "13999999999",
}'
INSERT INTO user_login VALUES (2, b);
                                                                  • 用户1绑定了手机号码，微信，QQ
                                                                  • 用户2只绑定了手机号码

                                                                  如果不用JSON存储登录信息

                                                                  CREATE TABLE user_login (
    user_id		BIGINT PRIMARY KEY,
    cellphone	VARCHAR(255),
    wechat		VARCHAR(255),
    QQ			VARCHAR(255)
);

                                                                  虽然用传统关系型的方式也可以完成相关数据的存储，但是存在两个问题。

                                                                  • 有些列可能是比较稀疏的，一些列可能大部分都是 NULL 值；
                                                                  • 如果业务迭代，需要新增一种登录类型，如微博登录，则需要添加新列，而 JSON 类型无此烦恼。

                                                                  # 获取JSON数据

                                                                  JSON_EXTRACT：用来从JSON数据中提取所需要的字段内容

                                                                  JSON_UNQUOTE：用来去除JSON值的双引号

                                                                  SELECT
    user_id,
    JSON_UNQUOTE(JSON_EXTRACT(login_info,"$.cellphone")) cellphone,
    JSON_UNQUOTE(JSON_EXTRACT(login_info,"$.wxchat")) wxchat
FROM user_login;

                                                                  每次写 JSON_EXTRACT、JSON_UNQUOTE 非常麻烦，MySQL 还提供了 ->> 表达式，和上述 SQL 效果完全一样

                                                                  SELECT
    user_id,
    login_info->>"$.cellphone" cellphone,
    login_info->>"$.wechat" wechat,
FROM user_login;

                                                                  当 JSON 数据量非常大，用户希望对 JSON 数据进行有效检索时，可以利用 MySQL 的函数索引功能对 JSON 中的某个字段进行索引。

                                                                  # 查询

                                                                  MySQL 8.0.17 版本开始支持 Multi-Valued Indexes，用于在 JSON 数组上创建索引

                                                                  通过函数 MEMBER OF、JSON_CONTAINS、JSON_OVERLAPS 来快速检索索引数据

                                                                  • MEMBER OF：用于在 MySQL 中判断某个值是否存在于 JSON 数组中的操作符
                                                                  • JSON_CONTAINS：用于检查一个 JSON 文档是否包含另一个 JSON 文档或值
                                                                  • JSON_OVERLAPS ：用于检查两个 JSON 文档是否有重叠部分
                                                                  ALTER TABLE UserTag
ADD INDEX idx_user_tags ((cast((userTags->"$") as unsigned array)));
                                                                  • userTags->"$"：表示从JSON中提取整个JSON数组

                                                                  查询常看电影的用户

                                                                  SELECT * FROM UserTag 
WHERE 10 MEMBER OF(userTags->"$")

                                                                  查询画像为 80 后，且常看电影的用户

                                                                  SELECT * FROM UserTag 
WHERE JSON_CONTAINS(userTags->"$", '[2,10]')

                                                                  查询画像为80 后或者90 后或者常看电影的用户

                                                                  SELECT * FROM UserTag 
WHERE JSON_OVERLAPS(userTags->"$", '[2,3,10]')

                                                                  # 总结

                                                                  JSON 类型是 MySQL 5.7 版本新增的数据类型，用好 JSON 数据类型可以有效解决很多业务中实际问题

                                                                  • 使用 JSON 数据类型，推荐用 MySQL 8.0.17 以上的版本，性能更好，同时也支持 Multi-Valued Indexes；
                                                                  • JSON 数据类型的好处是无须预先定义列，数据本身就具有很好的描述性；
                                                                  • 不要将有明显关系型的数据用 JSON 存储，如用户余额、用户姓名、用户身份证等，这些都是每个用户必须包含的数据；
                                                                  • JSON 数据类型推荐使用在不经常更新的静态数据存储。
